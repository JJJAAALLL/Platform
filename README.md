# zeutec_db_mock

Mock database + dashboard for the Zeutec central data infrastructure project.
This is a simulation — no real instrument data, everything is generated by python package FAKER.

---

## What this is

We're building a cloud database for Zeutec NIR/Vision instruments that flow data
across a supply chain (farmers → millers → traders → buyers → logistics).
This repo is the sandbox to figure out the architecture before touching anything real.

The database is event-centric — everything that happens (a scan, an order, a delivery,
a farm boundary update) is an **event** in a ledger. Events chain together via
`previous_event_id` to preserve the causal order per organisation.

## Database tables (quick reference)

| Table | What it stores |
|---|---|
| `organisation` | tenants in the system (FARMER, MILLER, TRADER, BUYER, LOGISTICS, LAB, ZEUTEC) |
| `farm_detail` | lat/lon, size in ha, polygon GeoJSON — only for scanning orgs |
| `silo` | individual silos per org, each with location + capacity |
| `org_material` | which materials (wheat, soy…) each org handles |
| `user_account` | users with roles scoped to their org type |
| `instrument` | physical Zeutec devices, owned by one org |
| `sensor_channel` | NIR-SWIR / VIS-RGB channels per instrument |
| `material` | wheat, soy, corn, milk powder, oil |
| `sample` | physical specimens with LIMS reference |
| `method` | acquisition recipe used during a scan |
| `calibration_model` | trained model: one analyte, one algorithm |
| `event_type` | lookup: MEASUREMENT, ORDER, DELIVERY, FARM_UPDATE, … |
| `event` | **the main table** — every reportable thing that happens |
| `measurement_data` | spectrum URI per scan (actual vectors live in S3, not here) |
| `measurement_image` | image URI for VISION AI instruments |
| `result` | predicted analyte values (Protein %, Moisture %, …) |
| `soft_data` | manual key-value annotations attached to events |

## Key rules baked into build.py

These are the consistency rules the seeder enforces (the DB schema can't enforce
cross-FK relationships on its own):

- operator on an event must belong to the **same org** as the event
- instrument on an event must belong to the **same org** as the event
- sensor channel must belong to the **instrument used in the event**
- operator role must match the event type (e.g. only DRIVERs do DELIVERY events)
- event types are restricted by org type (e.g. LOGISTICS can't do MEASUREMENTS)
- instruments only exist at FARMER / MILLER / LAB / ZEUTEC orgs
- `previous_event_id` always points to the previous event of the **same org**
- `soft_data.recorded_by` must belong to the same org as the event

All of these are checked at the end of `build.py`. If anything fails the DB is deleted.

## Making changes

**Add a new event type** — add it to `event_type` inserts in `build.py`,
update `EVENT_TYPES_BY_ORG_TYPE` and `ROLES_BY_EVENT_TYPE` dicts, add a sanity check.

**Add a column to a table** — edit the `SCHEMA` string in `build.py` and update
any relevant query in `db.py`. Rebuild with `python build.py`.

**Add a new dashboard page** — create `app/views/yourpage.py` with a `render()` function,
then add it to the `page` radio in `main.py`.

**Change the map style** — in `map_view.py`, find `folium.Map(... tiles="OpenStreetMap")`
and swap to `"CartoDB positron"` for a cleaner grey look or `"Esri WorldImagery"` for satellite.

**Scale up fake data** — edit the N_ constants at the top of `build.py`:
```python
N_EVENTS = 1000   # crank this up to stress test queries
```
